Agon decouples payment execution from settlement. Participants can make many payments first through signed commitments and settle them later onchain.

Each wallet registers once as a participant and keeps the same `participant_id` for the life of the deployment. Participants deposit allowlisted tokens into protocol balances. When one participant pays another, they create one permanent one-way payment channel for that token. The payer can lock funds to that channel if the relationship needs guaranteed payment capacity.

## Overview

The current protocol uses two signed message formats:

- `agon-cmt-v5` for one payment channel
- `agon-round-v4` for one cooperative clearing round

Those messages feed three settlement paths:

- direct settlement for one channel
- bundle settlement for several commitments to one payee
- cooperative clearing for several participants that want to settle together

## The Four Main Onchain Objects

| Object | What it stores | Why it exists |
| --- | --- | --- |
| `GlobalConfig` | Authority, fee settings, chain id, message domain, timelock policy | Defines the deployment and the signing domain |
| `TokenRegistry` | The allowlisted settlement tokens | Makes token support explicit instead of implicit |
| `ParticipantAccount` | Stable `participant_id`, balances, withdrawal state, inbound-lane policy | Gives every wallet one durable protocol identity |
| `ChannelState` | One directed lane's token, payer, payee, settled cumulative amount, locked funds, signer state | Tracks the durable relationship between one payer and one payee |

## Payment Flow

In the common case, an integration looks like this:

1. register the payer and payee as participants
2. deposit an allowlisted token
3. create a one-way payment channel
4. optionally lock funds to that channel
5. sign cumulative commitments offchain as usage grows
6. settle the newest valid state later

## What Stays Offchain

Agon keeps signed payment updates offchain until settlement. Most integrations store:

1. the latest signed `agon-cmt-v5` per lane
2. local usage or metering state
3. bundle assembly logic
4. cooperative round construction logic

Only the newest valid state needs to reach the chain.

## When To Use Each Settlement Mode

Use **latest commitment settlement** when one payer and one payee need the simplest possible flow.

Use **bundle settlement** when one payee has collected signed commitments from many payers in the same token and wants one transaction instead of many.

Use **cooperative clearing rounds** when several parties are willing to co-sign one shared round so that payments inside the group can cancel each other out before balances are updated.

## Why V4 Uses Permanent Identities And Permanent Channels

Agon intentionally uses stable identities and permanent channels. That means more long-lived onchain state, but it makes the protocol easier to work with:

1. participant ids do not recycle
2. offchain services can index users without worrying about identity reuse
3. channels do not close and reopen
4. replay protection is simpler
5. payment lanes stay stable for clients and operators

## Common Use Cases

Agon is a good fit for:

- one buyer paying one provider over time
- many buyers paying one provider
- operator hubs that sit between users and providers
- cooperative settlement between several known parties
