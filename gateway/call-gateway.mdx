description: "A ready-made reference client that exercises every gateway route with a real Solana wallet."

`call-gateway` is a reference client for the Agon Gateway. It loads a real Solana wallet, resolves routes against the live catalog, and drives the x402 exact and SIWX flows end-to-end. It is the fastest way to test the gateway without writing client code.

Two modes are provided:

- **`seed`** — loop over every catalog route, build a cheap request, and pay (or sign) automatically.
- **`call`** — hit one route with custom path params, query params, or a JSON body.

## Installation

```bash
git clone https://github.com/Agonx402/call-gateway
cd call-gateway
npm install
cp .env.example .env
```

Then set one of:

```ini
SOLANA_PRIVATE_KEY_BASE58=<base58-encoded 64-byte secret key>
# or
SOLANA_PRIVATE_KEY_JSON_PATH=/absolute/path/to/keypair.json
```

Recommended for reliable signature discovery and payment payload construction:

```ini
SOLANA_MAINNET_RPC_URL=<your mainnet RPC>
SOLANA_DEVNET_RPC_URL=<your devnet RPC>
```

<Warning>
The buyer wallet must be different from the gateway's `payTo` wallet. If they match, x402 exact SVM creates a self-transfer and CDP will reject it.
</Warning>

## Seed mode

Seed mode exercises every route in the catalog. It is useful for smoke-testing the gateway, verifying wallet balance, or regenerating benchmarks.

```bash
npm run seed
```

What seed mode does:

- Fetches `GET /v1/catalog`.
- Preflights one unpaid challenge per payment network and fails fast on self-pay misconfigurations.
- Builds low-cost request bodies for every route.
- Uses SIWX automatically for Tokens auth-only routes.
- Uses the standard x402 `fetch` flow automatically for paid routes.
- Picks the right RPC URL per cluster when building payment payloads.
- Prints a success/failure summary at the end.

### Seed flags

| Flag | Effect |
| --- | --- |
| `--path <gateway-path>` | Exact route path to seed. Repeatable. |
| `--cluster <mainnet\|devnet>` | Filter by cluster. Repeatable. |
| `--provider <alchemy\|helius\|tokens>` | Filter by provider. Repeatable. |
| `--surface <rpc\|das\|tokens>` | Filter by surface. Repeatable. |
| `--method <name>` | Filter by method. Repeatable. |
| `--delay-ms <number>` | Override delay between purchases. |
| `--retry-limit <number>` | Override retry count per attempt body. |
| `--help` | Show CLI help. |

Environment overrides:

```ini
SEED_DELAY_MS=750
SEED_RETRY_LIMIT=3
SEED_ONLY_PATHS=/v1/x402/solana/mainnet/alchemy/rpc/getTransaction,/v1/x402/tokens/health
```

### Seed examples

```bash
npm run seed
npm run seed -- --path /v1/x402/solana/mainnet/alchemy/rpc/getTransaction
npm run seed -- --provider alchemy --cluster mainnet --method getTransaction
npm run seed -- --surface das --provider helius
npm run seed -- --provider tokens --method assetDetail
```

## Call mode

Call mode hits a single route with custom inputs.

```bash
npm run call -- --path /v1/x402/tokens/health
```

What call mode does:

- Resolves the path against the live catalog.
- Supports templated catalog paths via repeated `--path-param key=value`.
- Supports repeated `--query key=value`.
- Accepts `--body` or `--body-file` for POST requests.
- Uses SIWX automatically for Tokens routes.
- Uses the standard x402 payment flow for paid routes.
- Prints the JSON response and any settlement transaction hash.

### Call flags

| Flag | Effect |
| --- | --- |
| `--path <gateway-path>` | **Required.** Concrete or templated catalog path. |
| `--http-method <GET\|POST\|HEAD>` | Override HTTP method. Defaults to the catalog method. |
| `--path-param <key=value>` | Replace a templated path segment. Repeatable. |
| `--query <key=value>` | Append a query parameter. Repeatable. |
| `--body <json>` | JSON request body for POST calls. |
| `--body-file <path>` | Read JSON request body from a file. |
| `--help` | Show CLI help. |

### Call examples

```bash
npm run call -- --path /v1/x402/tokens/health
npm run call -- \
  --path /v1/x402/tokens/assets/:assetId/variants \
  --path-param assetId=ethereum \
  --query liquidityTier=tier1
npm run call -- \
  --path /v1/x402/tokens/assets/search \
  --query q=btc --query limit=5
npm run call -- \
  --path /v1/x402/tokens/assets/market-snapshots \
  --body '{"mints":["So11111111111111111111111111111111111111112"]}'
npm run call -- \
  --path /v1/x402/solana/mainnet/alchemy/rpc/getBalance \
  --body '{"params":["86xCnPeV69n6t3DnyGvkKobf9FdN2H9oiVDdaMpo2MMY"]}'
```

## Under the hood

The CLI uses:

- [`@x402/core`](https://www.npmjs.com/package/@x402/core) — payment-required envelope decoding and retry logic.
- [`@x402/fetch`](https://www.npmjs.com/package/@x402/fetch) — `wrapFetchWithPayment` to attach payment headers automatically.
- [`@x402/extensions/sign-in-with-x`](https://www.npmjs.com/package/@x402/extensions) — SIWX payload construction and encoding.
- [`@x402/svm`](https://www.npmjs.com/package/@x402/svm) — SVM-side exact scheme and Solana mainnet/devnet CAIP-2 constants.
- [`@solana/kit`](https://www.npmjs.com/package/@solana/kit) — Solana signer from a 64-byte keypair.

Nothing about the CLI is gateway-specific beyond the default base URL. It is a reasonable starting point for a custom client — see [Integration guide](/gateway/integration-guide) for how to build one.
