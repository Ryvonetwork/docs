Agon currently uses two signed message types:

1. `agon-cmt-v5` for one payment channel
2. `agon-round-v4` for one cooperative clearing round

This page explains the exact fields and encoding rules for both.

## Common Encoding Rules

Both message formats start with:

1. a one-byte `kind`
2. a one-byte `version`
3. a 16-byte `message_domain`

### `message_domain`

`message_domain` is derived inside the program from:

- a fixed Agon domain tag
- the program id
- the chain id

That means the signed domain is deployment-specific.

### Compact Integers

Participant ids and cumulative amounts use compact unsigned integer encoding in the signed message. In the test helpers this is implemented as `encodeCompactU64`.

### Token ID

`token_id` is stored as a little-endian `u16` in both message types.

## `agon-cmt-v5`

This is the unilateral payment-channel message used by:

- `settle_individual`
- `settle_commitment_bundle`

### Layout

| Field | Meaning |
| --- | --- |
| `kind = 0x01` | Unilateral commitment message |
| `version = 0x05` | Current unilateral wire format |
| `message_domain` | Deployment-scoped replay boundary |
| `flags` | Optional field presence bits |
| `payer_id` | Channel payer participant id |
| `payee_id` | Channel payee participant id |
| `token_id` | Allowlisted settlement token |
| `committed_amount` | New cumulative target for the channel |
| `authorized_settler` | Optional delegated submitter; the payee may still settle |

### Flags

| Bit | Meaning |
| --- | --- |
| `0x01` | `authorized_settler` is present |

### Practical Meaning

`committed_amount` is cumulative, not incremental.

If the channel is already settled to `1_000_000` and the new message says `1_250_000`, then the settlement delta is `250_000`.

Operator or coordinator payment is not embedded in `agon-cmt-v5`. If a flow needs to compensate an operator, that payment should be expressed as its own ordinary lane commitment.

## `agon-round-v4`

This is the cooperative clearing message used by `settle_clearing_round`.

### Layout

| Field | Meaning |
| --- | --- |
| `kind = 0x02` | Cooperative clearing-round message |
| `version = 0x04` | V4 wire format |
| `message_domain` | Deployment-scoped replay boundary |
| `token_id` | Token shared by the entire round |
| `participant_count` | Number of signed participants in the roster |
| `participant_id` | Participant block owner |
| `entry_count` | Number of outgoing channel targets in that block |
| `payee_ref` | Index of the payee inside the signed roster |
| `target_cumulative` | New cumulative target for that channel |

### How To Read A Round

The round is organized by payer block.

Each block says:

1. which participant is the payer
2. how many outgoing channel targets that payer is advancing
3. for each target, which signed participant in the roster is the payee
4. what the new cumulative amount for that channel should be

`payee_ref` keeps the message compact by reusing the roster instead of repeating full payee ids for every edge.

## Which Message To Use

Use `agon-cmt-v5` when one payer is updating one channel.

Use `agon-round-v4` when several participants are willing to sign one shared message so that several channel updates can be settled together.
