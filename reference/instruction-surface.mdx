This page lists the instruction surface exactly as it exists in the current program.

Most integrations only need a small subset, but it is useful to know the full surface.

## Bootstrap And Admin

`initialize`  
Creates `GlobalConfig`, fixes the deployment chain id, derives the immutable message domain, and sets the initial authority flow.

`update_config`  
Updates mutable configuration such as fee recipient, fee basis points, or registration fee, and can nominate a pending authority.

`accept_config_authority`  
Completes the two-step config authority handoff.

`initialize_token_registry`  
Creates the singleton token registry after bootstrap.

`register_token`  
Adds one allowlisted token and creates its vault token account PDA.

`update_registry_authority`  
Nominates a pending registry authority.

`accept_registry_authority`  
Completes the two-step registry authority handoff.

## Participants

`initialize_participant`  
Registers one wallet as one permanent participant.

`update_inbound_channel_policy`  
Sets whether inbound channels are permissionless, consent-based, or disabled.

## Funding And Withdrawals

`deposit`  
Moves tokens from the user's token account into the protocol vault and credits participant balance.

`deposit_for`  
Lets one funder credit several participants in one transaction.

`request_withdrawal`  
Starts a timelocked withdrawal for one token and records the destination token account.

`cancel_withdrawal`  
Cancels a pending withdrawal and returns the amount to available participant balance.

`execute_withdrawal_timelocked`  
Transfers the withdrawal out of the vault once the timelock has expired.

## Channels

`create_channel`  
Creates one permanent one-way payment channel from payer to payee for one token.

`lock_channel_funds`  
Moves part of the payer's participant balance into channel-specific locked balance.

`request_unlock_channel_funds`  
Starts a timelocked request to return some locked channel balance to the payer's shared balance.

`execute_unlock_channel_funds`  
Executes that unlock after the timelock.

`request_update_channel_authorized_signer`  
Starts a timelocked channel-signer rotation.

`execute_update_channel_authorized_signer`  
Executes the signer rotation after the timelock.

## Settlement

`settle_individual`  
Settles one `agon-cmt-v5` message for one channel.

`settle_commitment_bundle`  
Settles many `agon-cmt-v5` messages in one transaction for one payee.

`settle_clearing_round`  
Settles one `agon-round-v4` message signed by a roster of participants and applies only the residual balance changes after the included payments cancel each other out.

## Most Integrations Need

If you are building a provider, gateway, or agent client, the smallest useful subset is usually:

1. `initialize_participant`
2. `deposit`
3. `create_channel`
4. optionally `lock_channel_funds`
5. `settle_individual` or `settle_commitment_bundle`

If you are building a cooperative settlement coordinator, add `settle_clearing_round`.
